Lab Submissions

Five labs that built the foundation for my projects — each with screenshots, a video walkthrough, and a personal reflection.

L1

Lab 1 — Basic Layout with Jetpack Compose

UI · Composable layouts, Rows, Columns & styling

Video Walkthrough

▶ Watch on YouTube

Screenshots

ComposeColumn / RowModifiers

Reflection: Lab 1 was my first real contact with Jetpack Compose, and it changed how I think about building screens. Instead of XML layouts, I described the UI declaratively with composable functions and arranged content using Column, Row, and Modifier. I learned how padding, spacing, and alignment work together, and how the live preview speeds up the design loop. This lab gave me the confidence to build the visual foundation that every later lab and my Jobsy app would stand on.

L2

Lab 2 — Interaction with Jetpack Compose

Interaction · State, buttons, text fields & user input

Video Walkthrough

▶ Watch on YouTube

Screenshots

rememberStateEvents

Reflection: This lab taught me how a Compose UI comes alive. I used state with remember and mutableStateOf so the screen reacts instantly when a user types or taps a button. Understanding recomposition — that the UI redraws when state changes — was the key insight. Handling text input and click events here directly prepared me for Jobsy's sign-in fields and the volunteer application form, where capturing and validating user input is essential.

Lab 4 — Navigation & ViewModel

Reflection: Lab 4 connected everything into a real multi-screen app. I learned to set up a NavHost with named routes, pass arguments between screens, and keep screen state in a ViewModel so it survives configuration changes like rotation. Project Submissions

Two projects that grew Jobsy from a navigable prototype into a full app with API, sensors, and cloud — all aligned to SDG 1: No Poverty.

P1

Project 1 — Jobsy (Lab 4 + Working Features)

Navigation-based app with functional screens & flows

Problem Statement

Many people in the community face difficulties finding urgent jobs, temporary work, or basic support such as food, groceries, and household items. Some don't need permanent employment but still need quick ways to earn income or receive essentials — while others want simple help with daily tasks, and some want to volunteer but don't know where. Jobsy connects these three groups in one place.

Lessons Learned

  • Planning a navigation graph early makes adding screens far easier later.
  • Keeping state in ViewModels avoids losing user input on rotation.
  • Breaking the UI into reusable components (like JobCard) keeps screens clean.
  • A clear problem statement keeps every feature focused on real user needs.
P2

Project 2 — Jobsy (Lab 5 + API + Sensors + Cloud)

2+ screens · OpenWeather API · GPS & location sensors · Firebase Cloud

Problem Statement

Building on Project 1, Jobsy now lets users browse community jobs from the cloud, check the weather for a job's location before applying, pick locations on a map using their device's GPS, apply with a guided form, and review their saved applications offline — directly supporting SDG 1 by making income and support opportunities more accessible.

Video Submission Report (VSR)

▶ Watch VSR ⌥ GitHub Repo

Screenshots

Firebase FirestoreRoomRetrofitGoogle MapsGPS

Key Features & Tech

  • Cloud (Firebase Firestore): shared, live list of community jobs.
  • API (OpenWeather + Retrofit): weather forecast for a job's location.
  • Sensors (Fused Location / GPS + Maps): map-based location picker using the device's current location.
  • Local storage (Room): saved volunteer applications, viewable offline.

Lessons Learned

  • Combining cloud, local DB, API, and sensors requires careful state and error handling.
  • Using coordinates instead of raw addresses makes weather and maps more accurate.
  • Putting validation and formatting in a dependency-free logic layer made the app easier to test and reason about.
  • Real APIs need API keys, permissions, and graceful fallbacks when the network fails.

Reflection & Learning Journey

Looking back from Lab 1 to Project 2, my growth wasn't just technical — it was about learning to build software that matters.

📈 Growth from Lab 1 to Project 2

  • Lab 1–2 · Foundations

    Learned to build and bring screens to life with Compose layouts and state.

  • Lab 3 · Design sense

    Adopted Material 3 to make apps look consistent and professional.

  • Lab 4 · Structure

    Connected screens with navigation and ViewModels — my first real app architecture.

  • Lab 5 · Data

    Persisted data locally with Room so information survives restarts.

  • Project 1–2 · Real product

    Combined it all, then added API, sensors, and cloud to ship Jobsy.

💪 Challenges I Overcame

  • Understanding recomposition and why my UI wasn't updating at first.
  • Passing arguments correctly between navigation routes.
  • Managing permissions for GPS and handling location on real devices.
  • Connecting Firebase and a real API key without breaking the build.
  • Keeping state consistent across cloud, Room, and the UI at the same time.

SDG Impact: Jobsy is my contribution to SDG 1 (No Poverty). By connecting people who need urgent income or essentials with those offering work, help, or volunteering, the app lowers the barrier to earning and receiving support. Building it taught me that technology has the most value when it's pointed at a genuine human need.
